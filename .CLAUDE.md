# Android Test Distribution Platform

Project-specific instructions for Claude Code working on this Android test distribution system.

## Project Overview

A web-based Android APK distribution platform for internal testing with:
- **Two applications**: Telecom (blue theme) and Partner (pink theme)
- **Version management**: Latest version + historical versions
- **Download tracking**: Statistics API for download counts
- **QR code generation**: Direct download QR codes for all versions
- **Simple deployment**: Static files served by nginx via Docker

## Technology Stack

- **Frontend**: Pure HTML/CSS/JavaScript (no frameworks)
- **QR Code generation**: qrcode.js (CDNJS)
- **Backend**: nginx (static file serving)
- **Stats API**: Node.js + Express
- **Deployment**: Docker Compose (local), rsync + SSH (production)
- **Production server**: 43.136.56.11:8080

## Development Workflow

### Local Development

```bash
# Start local development server
docker-compose up -d

# Access locally
open http://localhost:8080/android/telecom/
open http://localhost:8080/android/partner/

# View logs
docker-compose logs -f nginx
```

### Publishing New APK Versions

```bash
# Publish telecom app (interactive)
./publish_telecom.sh

# Publish partner app (interactive)
./publish_partner.sh

# Quick publish with auto-increment
./publish_telecom_auto.sh
```

### Deployment to Production

```bash
# Sync files to production server
rsync -avz -e "ssh -i /Volumes/macOS/documents/key/codex.pem" \
    data/apps/telecom/index.html \
    ubuntu@43.136.56.11:/home/ubuntu/app/android-test-distribution/data/apps/telecom/

# Verify deployment
ssh -i /Volumes/macOS/documents/key/codex.pem ubuntu@43.136.56.11 \
    "grep -c 'showQrCode' /home/ubuntu/app/android-test-distribution/data/apps/telecom/index.html"
```

## File Conventions

### JSON Data Format

**latest.json**:
```json
{
  "versionName": "5.10.52",
  "versionCode": "50100052",
  "publishTime": "2025-01-03 12:00:00",
  "fileName": "telecom-5.10.52-50100052.apk",
  "apkUrl": "./apk/telecom-5.10.52-50100052.apk",
  "fileSize": "50.2 MB",
  "checksum": {
    "md5": "...",
    "sha1": "..."
  }
}
```

**versions.json**:
```json
{
  "versions": [
    {
      "versionName": "5.10.51",
      "versionCode": "50100051",
      "publishTime": "2024-12-28 10:30:00",
      "fileName": "telecom-5.10.51-50100051.apk",
      "apkUrl": "./apk/telecom-5.10.51-50100051.apk",
      "fileSize": "50.1 MB"
    }
  ]
}
```

## Key Implementation Patterns

### QR Code Modal Pattern

All QR code functionality uses:
- Dynamic script loading for qrcode.js
- Click-outside-to-close functionality
- Consistent modal styling with backdrop blur
- Theme-aware colors (blue for telecom, pink for partner)

### Version Display Pattern

Historical versions are rendered using `displayHistoryVersions()`:
- Maps versions array to HTML
- Includes download and QR buttons
- Displays version metadata
- Handles empty version lists

### Theme System

- **Telecom**: Blue gradient (#00b4db to #0083b0)
- **Partner**: Pink gradient (#f5576c to #f093fb)
- Theme classes: `.partner-theme` for partner-specific overrides

## Common Tasks

### Adding a New Feature to Both Apps

1. Implement in `data/apps/telecom/index.html`
2. Copy implementation to `data/apps/partner/index.html`
3. Adjust colors for partner theme (pink variants)
4. Test locally with `docker-compose up`
5. Deploy both files to production
6. Verify on production server

### Updating APK Information

1. Place new APK in `data/apps/[app]/apk/`
2. Update `latest.json` with new version info
3. Add to `versions.json` (move old latest to versions)
4. Run checksum verification: `./verify_apk.sh`
5. Deploy files to server

### Modifying HTML/CSS/JS

- Keep HTML structure identical between apps (except theme)
- Use CSS classes for theming (avoid inline styles)
- JavaScript functions should be app-agnostic when possible
- Test on both telecom and partner apps

## Testing Checklist

## Important Notes

- **No build process**: Pure static HTML, no compilation needed
- **No authentication**: Internal use only, assume trusted users
- **No backend logic**: All validation is frontend-only
- **APK storage**: Files stored locally in `data/apps/[app]/apk/`
- **Version management**: Manual JSON file updates
- **Checksums**: Use MD5/SHA1 for APK integrity verification
